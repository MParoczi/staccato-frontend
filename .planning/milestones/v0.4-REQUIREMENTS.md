## Original Requirements

### NB-01 â€” Notebook Dashboard

Users can see all their notebooks on a dashboard page at `/app/notebooks` after logging in. Each notebook card shows cover color, title, instrument, and page size. An empty state is displayed when no notebooks exist.

**Acceptance:**
- `GET /notebooks` is called on mount and result rendered as notebook cards
- Dashboard renders with correct cover color swatch, title, instrument label, and page size per notebook
- Empty state UI is shown when the list is empty (not an error)
- Dashboard is the default landing after login (redirect from `/app`)

**Outcome:** Fully implemented. `NotebooksPage.tsx` renders responsive grid (`grid-cols-2 md:grid-cols-3`), skeleton cards while loading, and empty state with hint text. `/app` redirects to `/app/notebooks` via router index redirect.

---

### NB-02 â€” Create Notebook

Users can create a new notebook via a modal/dialog. Required fields: title. Optional fields: instrument (defaults to guitar), page size (defaults to user's `defaultPageSize`), cover color (palette picker), style preset (system presets: Classic, Colorful, Dark, Minimal, Pastel).

**Acceptance:**
- `POST /notebooks` sends correct payload on submit
- Newly created notebook appears on the dashboard immediately (cache-invalidated)
- Form validates: title is required; submit disabled while in flight
- Cover color palette renders selectable swatches

**Outcome:** Fully implemented. `NotebookFormDialog.tsx` create mode â€” `GET /instruments` fetches UUID for the first instrument; all form fields validated with zod + react-hook-form; TanStack Query invalidates `['notebooks']` on success.

---

### NB-03 â€” Rename & Reconfigure Notebook

Users can rename a notebook and change its cover color. Changes reflected on the dashboard immediately.

**Acceptance:**
- `PATCH /notebooks/{id}` sends updated title and/or cover color
- Dashboard card reflects new title and cover color without a full page reload
- TanStack Query cache invalidated after mutation success

**Outcome:** Fully implemented. `NotebookFormDialog.tsx` edit mode â€” pre-filled from `GET /notebooks/:id`; sends `PUT /notebooks/:id` with title + coverColor only (pageSize and instrumentId excluded â€” backend rejects immutable field updates).

---

### NB-04 â€” Delete Notebook

Users can delete a notebook. Deletion is permanent. A confirmation dialog is presented before the delete request is sent.

**Acceptance:**
- Confirmation dialog required before `DELETE /notebooks/{id}`
- Notebook removed from dashboard immediately after deletion
- Navigating to a deleted notebook's URL shows error state, not a crash

**Outcome:** Fully implemented. `DeleteNotebookDialog.tsx` displays notebook name and "This can't be undone" copy; loading spinner on confirm; Sonner error toast on failure; `PageErrorBoundary` catches any crash from a deleted notebook URL.

---

### NB-05 â€” Notebook Book View

When a user opens a notebook, the cover page renders first. An index page tab is accessible.

**Acceptance:**
- `/app/notebooks/{id}` renders notebook cover (title, instrument, cover color)
- Index page navigable from notebook view with empty state
- No console errors on first open with a new notebook
- Browser back button returns to dashboard

**Outcome:** Fully implemented. `NotebookPage.tsx` shell with tab navigation (Cover / Index) via NavLink. `NotebookCoverPage.tsx` full-bleed cover with luminance-based contrast (black/white title). `NotebookIndexPage.tsx` empty state. Navbar breadcrumb via `useMatch('/app/notebooks/:id/*')`.

---

### ERR-01 â€” Mutation Error Toasts

API mutation failures display exactly one Sonner toast. Toast messages are user-readable.

**Acceptance:**
- Create / rename / delete failures each produce exactly one Sonner toast
- Toast copy derived from RFC 7807 `detail` field or sensible fallback
- No unhandled promise rejections on mutation failure

**Outcome:** Fully implemented. `extractErrorMessage` inline per component (not a global handler) â€” prevents double-firing. 4xx errors toast correctly. T-09 (offline simulation) skipped â€” accepted gap with documented rationale.

---

### ERR-02 â€” Page Error Boundary

A `PageErrorBoundary` wraps all page-level routes. If a page crashes, user sees a recovery UI.

**Acceptance:**
- A deliberate throw inside a page renders the error boundary UI, not a white screen
- Error boundary displays a message and a link back to the dashboard
- Error boundary does not affect other routes

**Outcome:** Fully implemented. `PageErrorBoundary` class component with `getDerivedStateFromError` and `componentDidCatch`. 4Ă— `<PageErrorBoundary>` wrappings in `router.tsx` (NotebooksPage, ProfilePage, NotebookPage, book-view children).

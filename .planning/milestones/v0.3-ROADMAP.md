## Milestone Summary

**Key Decisions:**

- Profile page at `/app/profile`, accessible via navbar avatar dropdown — establishes the app's primary navigation pattern
- Avatar URL sourced from `UserResponse.avatarUrl` (backend CDN/blob URL); frontend displays directly without re-uploading
- `UserProfile.defaultInstrument: string` reconciled to `defaultInstrumentId: string | null` to match backend `Guid?`
- AppLayout introduces persistent navbar — all `/app/*` routes wrapped; foundational for all future phases (4–12)
- `authStore.updateUser` added for profile mutations to sync in-memory user state without triggering re-auth
- Language change calls `i18next.changeLanguage()` immediately on save — no page reload

**Issues Resolved:**

- `UserProfile` type mismatch (`defaultInstrument: string` vs backend `defaultInstrumentId: Guid?`) — reconciled in PLAN-01
- Profile mutations returning 405 Method Not Allowed — profileApi.ts methods/URLs corrected in fix commit (382246b)
- Account deletion returning 404 — endpoint URL corrected to `DELETE /users/me/deletion`
- Avatar showing `?` instead of initials — `getInitials` helper in Navbar.tsx fixed

**Issues Deferred:**

- Hungarian profile strings are `__HU_TODO__` stubs (25 keys) — full HU translation deferred to Phase 12 (Localization)
- Google OAuth users: Google photo set as `avatarUrl` at login time by backend; no frontend-side photo refresh on re-login

**Technical Debt Incurred:**

- `GET /instruments` endpoint assumed to exist; if it changes shape, `InstrumentOption` interface in profileApi.ts needs updating
- No unit tests added for ProfilePage or profileApi — integration via UAT only; unit test coverage deferred

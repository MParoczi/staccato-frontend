### Task 4.1: Add per-module AbortController PATCH in CanvasRoot.tsx

**File:** `src/features/lessons/canvas/components/CanvasRoot.tsx`
**Action:** Modify

**Step 1 — Add `patchAbortRefs` and the `patchLayout` helper:**

At the top of the `CanvasRoot` component body, after the existing refs, add:

```typescript
import { patchModuleLayout } from '../api/modulesApi'
import type { PatchModuleLayoutPayload } from '@/types'

// Inside CanvasRoot component body:
const patchAbortRefs = useRef<Map<string, AbortController>>(new Map())

function patchLayout(moduleId: string, payload: PatchModuleLayoutPayload) {
  // Abort any in-flight PATCH for this module before sending a new one
  patchAbortRefs.current.get(moduleId)?.abort()
  const controller = new AbortController()
  patchAbortRefs.current.set(moduleId, controller)

  patchModuleLayout(moduleId, payload, controller.signal).catch((err: unknown) => {
    // Axios represents AbortController cancellation as 'CanceledError' (note the US spelling)
    if (err instanceof Error && err.name === 'CanceledError') return
    toast.error(extractErrorMessage(err, 'Failed to save changes'))
  })
}
```

Key notes:
- `Map<string, AbortController>` keyed by `moduleId` gives each module its own abort lifecycle. Aborting module A does not affect an in-flight PATCH for module B.
- The `.catch` handler silently swallows `CanceledError` (the Axios name for AbortController cancellation) and only shows the toast for real network errors.
- The `useRef` (not `useState`) ensures the map is stable across renders and mutations to the map do not trigger re-renders.
- Do NOT use `useMutation` for PATCH layout. `useMutation` is for operations the user intentionally initiates with clear UI feedback (POST, DELETE). PATCH layout is a fire-and-forget side effect of drag/resize — using `useMutation` would add loading state UI and cache invalidation that causes visible flicker on every pointer event.

**Step 2 — Call `patchLayout` from drag commit:**

In `handleDragEnd`, after the `setLocalModules` call, add:

```typescript
function handleDragEnd({ active, delta }: DragEndEvent) {
  const module = localModules.find(m => m.id === String(active.id))
  if (!module) return

  const rawNewX = module.gridX + delta.x / CELL
  const rawNewY = module.gridY + delta.y / CELL
  const clampedX = Math.max(0, Math.min(config.maxCols - module.gridWidth, Math.round(rawNewX)))
  const clampedY = Math.max(0, Math.min(config.maxRows - module.gridHeight, Math.round(rawNewY)))

  // Only patch if position actually changed (avoid unnecessary requests)
  if (clampedX === module.gridX && clampedY === module.gridY) return

  setLocalModules(prev => prev.map(m =>
    m.id === module.id ? { ...m, gridX: clampedX, gridY: clampedY } : m
  ))

  // Persist to backend — abort any previous in-flight PATCH for this module
  patchLayout(module.id, { gridX: clampedX, gridY: clampedY })
}
```

**Step 3 — Call `patchLayout` from resize commit:**

In `handleResizeCommit`, after the `setLocalModules` call, add:

```typescript
function handleResizeCommit(
  moduleId: string,
  patch: { gridX: number; gridY: number; gridWidth: number; gridHeight: number }
) {
  setLocalModules(prev => prev.map(m =>
    m.id === moduleId ? { ...m, ...patch } : m
  ))

  // Persist to backend
  patchLayout(moduleId, patch)
}
```

**Step 4 — Call `patchLayout` from z-order changes:**

In `handleBringForward` and `handleSendBackward`:

```typescript
function handleBringForward(moduleId: string) {
  // Compute new zIndex BEFORE entering the state updater — keeps the updater pure
  // (React 19 Strict Mode double-invokes updaters in development; side effects inside
  // updaters would fire twice, causing two PATCH requests per z-order change)
  const current = localModules.find(m => m.id === moduleId)
  if (!current) return
  const newZIndex = current.zIndex + 1

  setLocalModules(prev => prev.map(m =>
    m.id === moduleId ? { ...m, zIndex: newZIndex } : m
  ))
  patchLayout(moduleId, { zIndex: newZIndex })
}

function handleSendBackward(moduleId: string) {
  const current = localModules.find(m => m.id === moduleId)
  if (!current) return
  const newZIndex = Math.max(0, current.zIndex - 1)

  setLocalModules(prev => prev.map(m =>
    m.id === moduleId ? { ...m, zIndex: newZIndex } : m
  ))
  patchLayout(moduleId, { zIndex: newZIndex })
}
```

Note: The new zIndex is computed from `localModules` (the current state snapshot) before calling `setLocalModules`. Since `localModules` is read directly (not via an updater function), it reflects the committed state at the time the z-order button is clicked — the correct value to send to the API. The updater function is kept pure (no side effects), which is safe under React 19 Strict Mode.

**CRITICAL — snapToGrid at ALL write paths:** Verify that every value sent to `patchLayout` is an integer:
- `gridX`, `gridY`: set by `Math.round()` in `handleDragEnd` and by `ResizeHandle.handlePointerUp` which also calls `Math.round()`
- `gridWidth`, `gridHeight`: set by `Math.round()` and enforced `>= minWidth/minHeight` in `ResizeHandle.handlePointerUp`
- `zIndex`: incremented/decremented by 1 from an integer starting value — always integer

Never send a fractional grid value to the API. The live preview during resize uses fractional values in `localModules`, but `handleResizeCommit` (which calls `patchLayout`) receives the already-snapped integer values from `ResizeHandle.handlePointerUp`.

**File:** `src/pages/LessonPage.tsx`
**Action:** Verify and fix if needed

**Check 1 — pageSize from notebook query:**

Open `LessonPage.tsx` (as modified in Plan 1). Confirm:
- `useQuery(['notebook', notebookId])` exists and calls `getNotebook(notebookId!)`
- `notebook?.pageSize` is cast to `NotebookPageSize` with a default of `'A4'`

'A4') as NotebookPageSize` is safe because the backend only produces these three values (per the swagger spec and the `NOTEBOOK_PAGE_SIZES` constant).

If the notebook query is not already added: add it now following the pattern from `Plan 1 Task 1.5`.

**Check 2 — `activePage.id` as `pageId`:**

In `LessonPage.tsx`, confirm that `CanvasRoot` receives `pageId={activePage.id}`. The `activePage` variable is:
```typescript
const activePage = pages?.[currentPage - 1] ?? null
```

The `CanvasRoot` is already conditionally rendered:
```typescript
{activePage && (
  <CanvasRoot pageId={activePage.id} pageSize={pageSize} ... />
)}
```

This ensures:
1. `CanvasRoot` only mounts when there is a valid page (no `undefined` pageId)
2. When the user navigates to a different page (`?page=2`), `activePage` changes, `pageId` changes, and the `useQuery(['modules', pageId])` in `CanvasRoot` fires for the new page

**Check 3 — modules query key:**

In `CanvasRoot.tsx`, verify the query:
```typescript

  queryKey: ['modules', pageId],
  queryFn: () => getModules(pageId),
  enabled: !!pageId,
})
```

The `enabled: !!pageId` guard prevents the query from firing when `pageId` is empty string or undefined. Since `CanvasRoot` only renders when `activePage` is non-null (guaranteed by the conditional in LessonPage), this guard is belt-and-suspenders.

**Check 4 — `localModules` sync on page navigation:**

When the user navigates from page 1 to page 2, React unmounts the `CanvasRoot` for page 1 and mounts a new one for page 2 (because `pageId` changes and the component is conditionally rendered). The new `CanvasRoot` starts with empty `localModules` and the query fetches fresh module data. This is correct behavior — no additional state reset is needed.

If for any reason `CanvasRoot` does NOT unmount on page change (e.g., if it is rendered outside the conditional), add a `key={pageId}` prop to force remount:
```typescript
<CanvasRoot key={activePage.id} pageId={activePage.id} pageSize={pageSize} ... />
```
